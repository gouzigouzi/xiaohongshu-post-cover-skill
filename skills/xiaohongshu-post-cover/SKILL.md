# Xiaohongshu Post Cover

## 目标

将用户提供的素材整理成适合小红书发布的中文内容。默认使用适量 emoji 提升可读性；图片是可选输出，用户未明确要求时先完成文案，再询问是否生成封面。

## 工作流

1. 提取主题、具体经历或信息、用户立场、目标读者，以及是否要求封面图。
2. 判断内容意图：经验分享、工具测评、避雷吐槽、教程总结、推荐种草、求助讨论或观点反思。
3. 输出适配该意图的文案，包括标题、正文、必要的互动结尾和话题标签。
4. 保持真实具体、轻松可读，不将普通分享强行改写成求助或营销内容。
5. 若用户没有明确要求图片，在文案完成后简短询问一次是否需要封面图。
6. 若用户要求生成图片，使用可用的图像生成能力制作竖版小红书封面；若当前无图像工具，则提供可直接用于生图的封面提示词。

## 内容意图

| 类型 | 判断线索 | 合适的结尾 |
|---|---|---|
| 经验分享 | 记录过程或个人感受 | 总结收获、感受或下一步。 |
| 工具测评 | 评价工具、应用、产品或工作流 | 给出优缺点及适用场景。 |
| 避雷吐槽 | 强调踩坑、限制或低效 | 总结风险与注意事项。 |
| 教程总结 | 分享步骤、方法、笔记或操作流程 | 给出清单、成果或复用提示。 |
| 推荐种草 | 表达明确喜爱或推荐意愿 | 说明适合谁以及推荐理由。 |
| 求助讨论 | 明确说“求助”“想问问大家”“有没有人用过”或征集经验 | 提出 1-3 个具体问题。 |
| 观点反思 | 讨论趋势、取舍或个人思考 | 以个人结论或开放思考收束。 |

当用户意图不够明确时，根据素材最明显的方向写作，不要默认选择“求助讨论”。

## 文案要求

- 使用简体中文，保持用户原本的立场与语气强度。
- 使用 emoji 作为段落锚点或要点标识，不要每句都添加。
- 优先使用短段落和可扫描结构。
- 标题具体、有信息量；有帮助时提供 2-4 个标题备选。
- 默认在文末提供 6-10 个相关标签，除非用户要求不加。
- 不虚构体验、结果、数据或产品能力。
- 不使用夸张种草腔或生硬广告语，除非用户明确需要商业宣传风格。

## 常用结构

```text
标题：<具体标题，按需带 0-1 个 emoji>

<一句进入主题的个人背景或结论>

<实际经历、步骤、优缺点或方法内容>

<与帖子意图匹配的总结 / 推荐 / 提醒 / 提问>

#标签 #标签 #标签
```

## 封面图规则

- 用户明确提出“配图”“封面”“图片”“生成图”等要求时，直接进入封面生成。
- 用户只要求文案时，先输出完整文案，再询问是否需要封面；不要自动生成。
- 用户说图片“可有可无”“到时候再说”时，同样先询问再生成。

询问示例：

```text
要不要我再生成一张小红书竖版封面图？我可以让主标题更醒目，并匹配这篇帖子的内容类型。
```

## 封面图提示词要点

生成封面时，提示词应包含：

- 使用场景：小红书竖版封面，优先 `3:4` 或 `4:5`。
- 文字：一个短主标题和一个可选副标题，要求简体中文清晰醒目。
- 视觉元素：必须与主题直接相关，例如浏览器界面、论文、学习桌面、产品物件或步骤清单。
- 情绪：匹配帖子类型，例如好奇、实用、复盘、避雷或讨论感。
- 限制：无水印、无无关品牌 logo、无密集小字、无虚假截图。

## 验证场景

输出前检查：

- 体验或测评帖没有在用户未表达求助时强行提问。
- 用户明确征集建议时，帖子包含具体、自然的互动问题。
- 教程或复盘帖以内容价值收束，不写成“大家都怎么做”。
- 用户未明确要求图片时，没有自行生成封面。
